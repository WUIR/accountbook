# Agent 项目说明

## 项目背景

当前项目为《Android 应用开发实践》课程设计，目标是设计并开发一款个人记账 App，暂定名称为“轻记账”。

产品面向学生、上班族和日常个人记账用户，核心目标是帮助用户快速记录收入和支出，查看本月收支情况，并通过月度预算提醒控制消费。

项目当前已有文档：

- `docs/阶段性开发任务.md`
- `docs/第一阶段mvp开发prd.md`

后续开发、补充文档或实现代码时，应优先参考以上两个文档。

## 当前默认目标

默认开发目标为第一阶段 MVP。

第一阶段只实现本地记账核心闭环：

新增账单 -> 保存到 SQLite -> 列表展示 -> 首页汇总 -> 预算提醒。

第一阶段交付重点：

- 本地记账功能可用。
- 数据关闭 App 后不丢失。
- 首页能展示本月收入、支出、结余。
- 默认分类和默认账户可用。
- 月度预算和预警可演示。

## 快速开发原则

项目采用“先核心，后扩展”的开发策略。

第一阶段应避免引入过多复杂功能，优先保证最小可用产品完整。

第一阶段不包含：

- 凭证上传。
- Excel/PDF 导出。
- 周统计。
- 复杂图表。
- 自定义分类。
- 自定义账户。
- 分类预算。
- 信用卡账期管理。
- 云同步。
- 登录注册。
- 账单筛选、编辑、删除。

以上功能放入后续阶段。

## 技术约束

结合上一级目录中的课程实验内容，项目优先使用 Android 原生技术和已学习过的库。

### 编程语言、框架、架构模式

- 编程语言：Java。
- UI 框架：Android 原生 View/XML 布局。
- 页面组织：优先使用一个主 Activity + 多个 Fragment；简单页面也允许使用独立 Activity。
- 数据访问：使用 `SQLiteOpenHelper` 封装数据库建表、初始化和 CRUD。
- 本地配置：使用 `SharedPreferences`。
- 架构模式：采用简单分层结构，至少区分 UI 层、数据访问层、实体模型层。
- UI 层负责界面展示和用户交互，不直接拼接复杂 SQL。
- 数据访问层负责 SQLite 读写和查询聚合。
- 实体模型层负责账单、分类、账户等数据对象。

### 允许的第三方库

第一阶段默认不主动新增第三方依赖。

允许使用的库仅限当前课程实验中已参考过或后续阶段明确需要的库：

- Glide：后续用于凭证图片加载和展示。
- Gson：后续用于 JSON 备份、导入导出。
- OkHttp：后续用于云同步或网络备份预留。

如第一阶段没有明确需求，不应引入以上库。

### 禁止的技术方案

- 不使用 Kotlin 重写项目。
- 不使用 Jetpack Compose。
- 不使用 Room 替代 SQLiteOpenHelper，除非用户明确要求。
- 不引入大型 MVVM/MVI 框架。
- 不引入数据库 ORM 框架。
- 不引入重量级 Excel 库作为第一阶段实现方案。
- 不在第一阶段实现云同步、登录注册、复杂图表、PDF/Excel 导出。
- 不为了局部功能重构整个项目结构。
- 不主动新增依赖，除非当前需求明确需要且用户同意。

### 推荐技术

- Java：主要开发语言。
- SQLiteOpenHelper：本地数据库。
- SharedPreferences：保存预算金额、预警开关等轻量配置。
- ListView + BaseAdapter：展示账单、分类、账户列表。
- Activity + Fragment：组织页面结构。
- Thread + Handler：后续处理统计、导出等耗时任务。
- Glide：后续用于凭证图片显示。
- Gson：后续用于 JSON 备份或导入导出。
- OkHttp：后续预留网络同步或备份能力。
- PdfDocument：后续用于 PDF 账单导出。
- Canvas：后续用于绘制简单统计图表。

第一阶段不应优先引入重型框架或复杂第三方库。

## 开发规范

### 文件结构

如果后续创建 Android 工程，建议按以下结构组织 Java 代码：

- `activity`：Activity 页面。
- `fragment`：Fragment 页面。
- `adapter`：ListView/RecyclerView 适配器。
- `db`：SQLiteOpenHelper 和 DAO 类。
- `model`：实体模型类。
- `util`：日期、金额格式化、校验等工具类。

资源文件建议：

- `res/layout`：页面和列表项布局。
- `res/drawable`：图标、背景、形状资源。
- `res/mipmap`：启动图标。
- `res/values`：颜色、字符串、尺寸、样式。

第一阶段不要为了目录完整性创建空目录或空类。只有在功能需要时再新增文件。

### 命名规范

- Java 类名使用大驼峰，例如 `BillRecord`、`AccountDao`、`HomeFragment`。
- 方法名和变量名使用小驼峰，例如 `saveBillRecord`、`monthlyExpense`。
- 常量使用全大写下划线，例如 `TYPE_EXPENSE`、`DEFAULT_BUDGET_WARN_PERCENT`。
- XML 布局文件使用小写下划线，例如 `fragment_home.xml`、`item_bill_record.xml`。
- 控件 id 使用语义化小驼峰或下划线风格，但同一项目内保持一致，例如 `tvMonthlyIncome` 或 `tv_monthly_income`。
- 数据库表名和字段名使用小写下划线，例如 `bill_record`、`record_date`。

### 函数长度限制

- 单个普通函数建议不超过 60 行。
- UI 事件处理函数建议不超过 40 行。
- 超过限制时，优先提取为私有方法，但不要过度拆分。
- 数据库查询、校验、格式化、UI 刷新应拆成职责清晰的方法。

### 类职责限制

- Activity/Fragment 只负责页面初始化、事件绑定、状态刷新和跳转。
- Adapter 只负责列表项渲染，不写数据库逻辑。
- SQLiteOpenHelper 负责建表、升级和初始化默认数据。
- DAO 类负责具体数据读写。
- Model 类只保存数据字段和简单访问方法。
- Util 类只放无状态工具函数。
- 单个类不要同时承担 UI、数据库、业务计算和文件导出职责。

### 注释规范

- 注释应解释业务规则、边界条件或不直观的实现原因。
- 不写重复代码含义的空泛注释。
- 数据库表、关键字段、预算预警规则、账户余额联动规则应有简短注释。
- 临时调试注释和无用 TODO 不应提交到最终版本。

### 测试要求

第一阶段至少验证以下场景：

- 新增收入记录后，账单列表出现记录，账户余额增加。
- 新增支出记录后，账单列表出现记录，账户余额减少。
- App 关闭后重新打开，账单仍存在。
- 首页本月收入、本月支出、本月结余计算正确。
- 金额为空、为 0、为负数时不能保存。
- 收入分类和支出分类不会混用。
- 月度预算保存后重新打开仍存在。
- 支出达到 80% 和 100% 时预算提醒状态正确。

如项目没有自动化测试条件，应至少进行手工测试，并在交付说明中列出已验证项。

## 第一阶段数据设计

### bill_record 表

用途：保存账单记录。

字段：

- `id`：INTEGER，主键，自增。
- `type`：TEXT，收入或支出，值为 `income` / `expense`。
- `amount`：REAL，金额。
- `category_id`：INTEGER，分类 ID。
- `account_id`：INTEGER，账户 ID。
- `record_date`：TEXT，账单日期，格式 `yyyy-MM-dd`。
- `remark`：TEXT，备注。
- `create_time`：INTEGER，创建时间戳。

### category 表

用途：保存收支分类。

字段：

- `id`：INTEGER，主键，自增。
- `name`：TEXT，分类名称。
- `type`：TEXT，收入分类或支出分类。
- `sort_order`：INTEGER，排序值。

默认支出分类：

- 餐饮。
- 交通。
- 购物。
- 娱乐。
- 学习。
- 其他。

默认收入分类：

- 生活费。
- 工资。
- 兼职。
- 红包。
- 其他。

### account 表

用途：保存账户。

字段：

- `id`：INTEGER，主键，自增。
- `name`：TEXT，账户名称。
- `account_type`：TEXT，账户类型，值为 `cash` / `bank_card` / `credit_card`。
- `balance`：REAL，账户余额。

默认账户：

- 现金，余额 0。
- 银行卡，余额 0。
- 信用卡，余额 0。

### SharedPreferences

用途：保存预算配置。

字段：

- `monthly_budget`：月度预算金额。
- `budget_warn_enabled`：是否开启预算提醒。

## 核心业务规则

账单保存规则：

- 金额不能为空。
- 金额必须大于 0。
- 必须选择收支类型。
- 必须选择分类。
- 必须选择账户。
- 日期默认当天。
- 备注可以为空。

账户余额规则：

- 新增收入时，对应账户余额增加。
- 新增支出时，对应账户余额减少。

首页统计规则：

- 本月收入 = 当前自然月所有收入记录金额之和。
- 本月支出 = 当前自然月所有支出记录金额之和。
- 本月结余 = 本月收入 - 本月支出。

预算规则：

- 预算只统计支出，不统计收入。
- 预算周期为自然月。
- 预算进度 = 本月支出 / 月度预算。
- 支出低于预算 80%：正常状态。
- 支出达到预算 80% 且低于 100%：提醒“本月预算已使用较多”。
- 支出达到或超过预算 100%：提醒“本月预算已超出”。

## 验收标准

第一阶段完成后，应满足：

- App 启动后能进入首页。
- 首页能展示本月收入、本月支出、本月结余。
- 用户能新增一笔支出记录。
- 用户能新增一笔收入记录。
- 账单能保存到 SQLite。
- 关闭 App 后重新打开，账单仍存在。
- 最近账单列表能正确展示记录。
- 默认分类能正常选择。
- 默认账户能正常选择。
- 新增账单后账户余额能正确变化。
- 用户能设置月度预算。
- 首页能显示预算进度。
- 支出达到 80% 时能显示预算提醒。
- 支出达到或超过 100% 时能显示超额预警。
- 金额为空时不能保存。
- 金额为 0 或负数时不能保存。
- 收入和支出分类不会混用。
